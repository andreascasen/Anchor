# Deployment Plan for Anchor

## Context
Anchor is a personal Bun + Hono + SQLite API that needs to be publicly accessible (for AI agents, scripts, etc.), support a persistent SQLite database, and sync a private GitHub-hosted Obsidian vault. It will eventually run scheduled jobs. The goal is a cloud deployment that's always-on, low-cost (fixed price), and auto-deploys on git push.

## Platform: Fly.io

**Why Fly.io:**
- Fixed machine pricing (~$2–3/month for `shared-cpu-1x` with 512MB RAM)
- Free tier may cover it entirely (3 small VMs + 3GB storage included)
- Persistent volumes for SQLite (critical)
- Always-on with `auto_stop_machines = "off"`
- Custom domains supported
- Auto-deploy via GitHub Action (~10 lines of YAML)
- Good Bun/Docker support

## Architecture

```
[GitHub: Anchor repo]
        |
        | push to main → GitHub Action → flyctl deploy
        ↓
[Fly.io Machine: shared-cpu-1x, 512MB RAM]
  ├── Bun HTTP server (Hono)        ← tasks API + sync endpoint
  ├── Persistent volume at /data    ← brain.db (SQLite)
  └── Obsidian vault at /data/vault ← cloned from private GitHub repo on first run
```

## What Needs to Be Built

### 1. `Dockerfile`
Simple Bun container — install deps, copy source, expose port, run `bun src/index.ts`.

```dockerfile
FROM oven/bun:1
WORKDIR /app
COPY package.json bun.lockb ./
RUN bun install --frozen-lockfile
COPY . .
EXPOSE 3000
CMD ["bun", "src/index.ts"]
```

### 2. `fly.toml`
Fly.io config file — defines machine size, port, volume mount, and disables auto-stop.

Key settings:
- `auto_stop_machines = "off"` — always-on, no cold starts
- Volume mounted at `/data`
- Internal port `3000`

### 3. Vault clone logic (`src/dataSources/obsidian.ts`)
`syncAndIndex()` currently assumes the vault exists and does `git pull`. On a fresh container the vault won't be there. Update the logic to:
- If `OBSIDIAN_VAULT_PATH` doesn't exist → `git clone` using the GitHub PAT
- If it exists → `git pull` as today

### 4. GitHub Action (`.github/workflows/deploy.yml`)
Triggers on push to `main`, runs `flyctl deploy --remote-only`.
Requires `FLY_API_TOKEN` secret in the GitHub repo.

### 5. Environment variables (set via `fly secrets set`)
| Var | Value |
|---|---|
| `PORT` | `3000` |
| `API_SECRET` | your API key |
| `DB_DIRECTORY` | `/data` |
| `OBSIDIAN_VAULT_PATH` | `/data/vault` |
| `GITHUB_VAULT_TOKEN` | GitHub PAT with repo read access |
| `GITHUB_VAULT_REPO` | `https://github.com/user/vault.git` |

### 6. Custom domain
- Add domain in Fly.io dashboard
- Create a `CNAME` record pointing to the Fly-provided hostname

## Critical Files to Create/Modify
- `Dockerfile` — new file
- `fly.toml` — new file (generated by `fly launch`, then customized)
- `.github/workflows/deploy.yml` — new file
- `src/dataSources/obsidian.ts` — modify `syncAndIndex()` to clone if vault missing
- `src/env.ts` — add `GITHUB_VAULT_TOKEN` and `GITHUB_VAULT_REPO` env vars

## One-time Setup Steps (outside of code)
1. Install `flyctl` CLI and `fly auth login`
2. `fly launch` in the repo (generates `fly.toml`, creates the app)
3. `fly volumes create anchor_data --size 1` (persistent volume for SQLite + vault)
4. `fly secrets set API_SECRET=... GITHUB_VAULT_TOKEN=... GITHUB_VAULT_REPO=...`
5. Add `FLY_API_TOKEN` secret to GitHub repo settings
6. Configure custom domain in Fly.io dashboard + update DNS

## Verification
1. Deploy via `fly deploy` — check logs, server starts on port 3000
2. `GET /api/tasks` with correct `x-api-key` → returns task list
3. `GET /api/sync` → vault clones successfully, notes indexed
4. Restart machine (`fly machine restart`) → SQLite data persists
5. Push a code change to `main` → GitHub Action redeploys automatically
6. Custom domain resolves and API is reachable
